# USE-MODES-2026-04-20 — GeneralStaff registration modes

**Date:** 2026-04-20 afternoon
**Context:** Captured after Ray raised registering three new
projects (Devforge, catalogDNA, Retrogaze) with very different
work shapes, and Claude observed that two of the three didn't fit
the existing "bot-primary" pattern. The observation generalized
into this mode taxonomy.

---

## Core insight

**GeneralStaff's value isn't only automation.** It's a project
discipline surface with four artifacts:

- `state/<id>/tasks.json` — canonical prioritized to-do list
- `state/<id>/MISSION.md` — scope boundary (what's in / what's out)
- `projects.yaml` §`hands_off` — enforced no-touch list
- `state/<id>/PROGRESS.jsonl` — audit log (Hard Rule #9)

The **dispatcher** is the component that runs bot cycles. The
**discipline** is what those four artifacts provide. A project can
benefit from the discipline without ever invoking the dispatcher
for a single cycle.

That reframing opens up at least three registration modes, each
appropriate for different project shapes.

---

## Mode A — Bot-primary

**Pattern:** Dispatcher runs bot cycles autonomously. Engineer
does the work; verification gate + reviewer adjudicate; hands_off
blocks sensitive paths. Cycle outcomes land on `bot/work` (or
`bot/creative-drafts`). Ray reviews + merges (or auto-merge kicks
in after 5 clean cycles per Hard Rule #4).

**Characteristics:**
- `engineer_command` produces a git diff the reviewer can
  structurally parse
- Most tasks are bot-pickable (`interactive_only: false`)
- Verification command is a test/lint gate that Boolean-decides
- Reviewer evaluates scope drift + hands_off + evidence-for-task
- Cycle time is minutes to tens of minutes
- Automation is the *default*; interactive work is the exception

**Examples (current fleet as of 2026-04-20):**
- generalstaff (dogfood — GS managing GS)
- gamr (Next.js + React SaaS, engineer_provider: aider)
- bookfinder-general (Python MCP tool, mostly Mode A, but
  creative_work_allowed=true carves out Mode A' for creative work)
- raybrain Phase 1 (5 tasks in one 27-min session)

**Fit indicators (use Mode A when):**
- The project has a clear roadmap + existing test suite
- Most upcoming work is correctness-typed (types, tests, refactors,
  docstrings, bug fixes with minimal-repro tests)
- Strategic decisions (product direction, UX, algorithms) are
  already scoped or stay with Ray via clear hands_off boundaries
- Ray wants to offload execution so he can focus elsewhere

---

## Mode B — Interactive-primary, GS as discipline layer

**Pattern:** The dispatcher may never run bot cycles (or runs
them only on a very narrow bot-safe slice). The project benefits
from GS's structural surface: task list, mission scope, hands_off
enforcement, audit log. All four artifacts hold; automation is
opt-in per task.

**Characteristics:**
- Most tasks are `interactive_only: true` at queue time (by
  default, not by accident)
- The `engineer_command` may still exist, but the dispatcher is
  rarely invoked
- The audit log captures both *interactive* edits (via CLI task
  done/add/rm) and *bot* edits (via cycle events), so PROGRESS.jsonl
  remains authoritative even when nothing automated happened
- Ray uses `generalstaff task list --project=<id>` as his canonical
  "what next" reference; `generalstaff task done` records
  completion provenance in the audit log
- A small subset of tasks may eventually opt into bot-pickable —
  usually mechanical ones the audit surfaces as bot-safe (e.g.
  type hints on a non-load-bearing module)

**Examples (proposed 2026-04-20):**
- catalogdna (relaxation-authorized 2026-04-20-catalogdna;
  vault-finalization + user-facing content stays with Ray,
  discipline layer organizes the rest)

**Fit indicators (use Mode B when):**
- The project has real users (user-facing changes need Ray's eye)
- The project's creative/taste surface is large relative to its
  correctness-work surface
- Ray has parallel interactive work at the project and benefits
  from structure-without-automation
- Registering as Mode A would require so many `interactive_only`
  flags that automation never fires anyway

**Degenerate case:** If EVERY task is interactive_only and no
hands_off enforcement is needed, the project is just a normal repo
— don't register it. Mode B's threshold is *meaningful* discipline:
a real MISSION scope, a real hands_off, a tasks.json that's used.

## Mode C — Binary-output pipeline with approve/reject

**Pattern:** Engineer_command runs a pipeline that produces
*binary artifacts* (images, audio, sprites, data files) rather
than a code diff the reviewer can parse. Cycle commits the
artifacts to a drafts branch; verification checks object-level
properties (file exists, perceptual hash threshold, size bounds,
etc.); reviewer is skipped (same rationale as gs-279 creative
cycles — scope-drift doesn't translate to binary output); Ray
approves or rejects visually.

**Characteristics:**
- `engineer_command` invokes a domain pipeline (sprite generator,
  audio render, dataset build, etc.) instead of a coding agent
- Output lands on a dedicated branch (mirrors
  `bot/creative-drafts` from gs-279 but for non-prose artifacts)
- Verification checks artifact-level properties, not code-level
  properties
- Reviewer is skipped by default (same as creative cycles)
- Human review is visual / perceptual, not code-reading
- Approval cadence matches how creative work actually gets
  reviewed (spritesheet-at-a-time, not diff-at-a-time)

**Examples (proposed 2026-04-20):**
- Retrogaze sprite quality pass (engineer runs the Retrogaze
  sprite pipeline, commits a spritesheet, Ray visually approves
  or sends it back with notes)

**Fit indicators (use Mode C when):**
- The project has an existing pipeline whose output is
  non-textual
- The output quality can only be judged perceptually, not
  programmatically
- There's value in running the pipeline regularly (nightly
  quality passes, regeneration-on-config-change, etc.) with
  audit-trail continuity

**Implementation status:** Mode C is a proposed pattern as of
2026-04-20. The infrastructure needed beyond current GS:
- Verification command semantics for non-test gates (file-exists
  + perceptual-hash variants). May be project-specific
  verification commands for now; a generic "artifact gate" is a
  Phase 8+ concern.
- Reviewer-skip path is already there (gs-279 skip-when-creative).
  Could generalize to skip-when-binary.
- Merge flow for binary artifacts — auto_merge semantics differ
  because the approve/reject is visual, not automated.

## Cross-mode invariants

Regardless of mode, these hold:

- **Hard Rule #5 (mandatory hands_off):** every registered project
  has a non-empty hands_off list.
- **Hard Rule #7 (code ownership):** bot only ever pushes to
  `bot/work` (or mode-specific drafts branches), never to the
  primary branch.
- **Hard Rule #9 (open audit log):** PROGRESS.jsonl is the
  authoritative record, regardless of whether work was bot-
  driven, interactive-with-CLI, or pipeline-produced.
- **MISSION.md scope boundary:** out-of-scope work is
  out-of-scope across modes; the document is the project's
  scope contract.
- **Ray's taste authority:** product direction, user-facing
  copy, algorithm choices, aesthetic decisions stay with Ray.
  Mode choice doesn't change that; it just changes how the
  bot/discipline helps him execute against the scope he sets.

## Mode selection cheat-sheet

| Question | A | B | C |
|---|---|---|---|
| Does the project produce a git diff as output? | Yes | Mixed | No (binary) |
| Are tests the verification gate? | Yes | N/A | Artifact-level checks |
| Are most tasks bot-pickable? | Yes | No | Yes (but skip reviewer) |
| Does the project have real users? | Maybe | Often yes | Often yes |
| Does Ray need to approve each output? | No (verified → auto-mergeable after 5) | N/A (he does the work) | Yes (visual) |
| Primary value of GS? | Automation | Discipline | Pipeline + audit |

---

## Mode-A-to-B migration path

A Mode A project can drift toward Mode B if its creative/taste
surface grows (e.g. gets paying users). The transition is:

1. Tighten hands_off (add more paths)
2. Flag more tasks `interactive_only: true` at queue time
3. Eventually the ratio flips and it's Mode B in practice
4. Update MISSION.md to reflect the new scope boundary
5. No formal "mode change" event needed — the artifacts adapt

Reverse direction (B → A) works the same way, loosening instead.

## Not yet addressed by this doc

- **Multi-repo projects** (e.g. a project that spans a frontend
  repo + backend repo). Current GS assumes one repo per project.
- **Cross-project coordination** (e.g. a change in catalogdna that
  needs a matching change in a catalogdna client library). Each
  is its own project registration; cross-project sync is manual.
- **Mode mixing within a single project** (e.g. Mode A for
  backend tasks + Mode C for generated sprite assets). Possible
  in principle — create two projects pointing at subdirs of the
  same repo — but we don't have production experience with this.
  Flag as Phase 9+ concern.
- **Mode C verification semantics** beyond file-exists. The
  artifact-quality question is genuinely hard and deserves its
  own design pass before Retrogaze's sprite pass lands in
  production use.

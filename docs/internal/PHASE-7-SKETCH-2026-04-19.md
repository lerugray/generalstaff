# Phase 7 sketch (2026-04-19)

**Status:** Design sketch. NOT approved for implementation yet. Lives
here so Ray (and a future Claude session) can return to it post-
Thursday when there's fresh quota to validate quality tradeoffs.

**Motivation captured 2026-04-19 evening:** autonomous bot cycles
currently invoke `claude -p --dangerously-skip-permissions` as the
engineer, which consumes Ray's Claude subscription quota. Scaling the
fleet or running dense bot sessions burns through the weekly cap fast.
The reviewer already routes to OpenRouter Qwen3 Coder (~$0.02/session,
paid per token) and Ollama (free, local). The engineer is the heavier
token spend per cycle. If the engineer could also route to
non-subscription providers for bulk scaffolding work, subscription
quota would stretch considerably further.

## What changes

**Today:** engineer is hardcoded to whatever `engineer_command.sh`
contains per-project. Every gamr / raybrain / dogfood cycle runs
`claude -p`. One provider.

**Phase 7:** engineer is pluggable. Projects pick from a registry
of engineer providers via a new field in `projects.yaml`:

```yaml
- id: gamr
  engineer_provider: openrouter-aider      # or claude, opencode, etc.
  engineer_model: qwen/qwen3-coder-plus     # optional; default per provider
```

Existing projects without `engineer_provider` keep the current
`claude` default. Migration is strictly opt-in per project. Hard
Rule 8 already documents BYOK; Phase 7 makes the engineer half of
the agent stack BYOK-friendly in the same sense the reviewer
already is.

## Candidate providers

### Aider (recommended starting point)

- Mature OSS agentic CLI: [aider-chat on PyPI](https://pypi.org/project/aider-chat/)
- First-class OpenRouter support via `--openai-api-base` and
  `--model openrouter/*`
- Handles repo context, git commits, test runs natively
- Uses SEARCH/REPLACE edit-block protocol (different from
  Claude Code's autonomous-edit style)
- Python-based, `pip install aider-chat`; cross-platform

### OpenCode

- Per Ray's global CLAUDE.md routing rules: free-tier cloud models
  (big-pickle, qwen3.6-plus-free, nemotron-3-super-free, etc.)
- Already installed on Ray's machine based on existing routing
  conventions
- Autonomous-ish capability; less polished than aider for multi-
  file edits
- Tertiary option if aider doesn't fit

### Custom OpenRouter wrapper

- Thin TypeScript CLI that takes a prompt, runs a tool-using agent
  loop against OpenRouter Qwen3 Coder Plus / Flash
- Maximum control; most work; longest to ship
- Deferred until after aider + opencode have been evaluated

## API surface

### projects.yaml

```yaml

  path: /home/ray/myproject
  engineer_provider: claude              # default; current behavior
  # OR
  engineer_provider: aider
  engineer_model: qwen/qwen3-coder-plus
  # OR
  engineer_provider: opencode
  engineer_model: opencode/qwen3.6-plus-free
```

### Environment variables

- `OPENROUTER_API_KEY` already wired for the reviewer; aider reads
  the same variable natively.
- No new secrets. Provider plumbing mirrors the existing reviewer
  pattern in `src/reviewer.ts`.

### engineer_command.sh (per project)

Current convention is each project ships its own `engineer_command.sh`
with `claude -p --dangerously-skip-permissions`. Phase 7 lifts this
into a GS-level dispatcher function that emits the right command for
the chosen provider:

```bash
# Generated by GS at session start, based on projects.yaml:
#   engineer_provider: aider
#   engineer_model: qwen/qwen3-coder-plus
aider \
  --model openrouter/qwen/qwen3-coder-plus \
  --yes \
  --no-auto-commits \
  --message-file "$PROMPT_FILE"
```

GS's `src/engineer.ts` (new file, or extension of existing) owns the
translation from provider name to shell command. Projects keep their
own `engineer_command.sh` as an override for advanced use cases;
default behavior comes from GS.

## Task-prompt adaptation

Current GS task specs assume Claude Code's autonomous-agent behavior:
"read the repo, figure out the codebase, make the changes, run the
tests." Aider and opencode expect different input shapes:

- **Aider** wants explicit file paths or auto-discovery via
  `--auto-gitignore`. Best with shorter, edit-focused prompts.
- **OpenCode** prompts are more chat-like; autonomous edit protocol
  varies by mode.

Phase 7 needs a **prompt adaptation layer** in `src/engineer.ts` that
translates a GS task spec into the target provider's expected format.
Shape:

```typescript
function buildEngineerInvocation(
  task: GreenfieldTask,
  project: Project,
  provider: EngineerProvider
): { command: string; args: string[]; stdin?: string } {
  switch (provider) {
    case 'claude':  return buildClaudeInvocation(task, project);
    case 'aider':   return buildAiderInvocation(task, project);
    case 'opencode': return buildOpenCodeInvocation(task, project);
  }
}
```

The Claude path stays byte-identical to current behavior. The aider
and opencode paths are new.

## Quality validation plan

Before recommending OpenRouter engineers for bulk use, measure on a
controlled benchmark:

1. **Pick 10 already-shipped gamr tasks** from gamr-001..035. Tasks
   with clear pass/fail verification (tests pass, typecheck, a11y
   checks) are ideal.
2. **Re-run them against aider + OpenRouter Qwen3 Coder Plus.**
   Dispatcher picks from a fresh bot/work branch in a disposable
   worktree so master isn't touched.
3. **Compare outcomes:** verified vs rolled-back, diff size, diff
   correctness (manual spot-check), cycle duration.
4. **Publish the results** in `PHASE-7-BENCHMARK-<date>.md` for
   transparency.

Acceptance criteria for rolling out OpenRouter engineer as default
for "bulk scaffolding" work:
- **Verified rate ≥ 70%** (matches gamr-001..035's observed rate
  under Claude engineer).
- **Mean cycle duration ≤ 1.5x** the Claude-engineer baseline.
- **No net regression** in hands-off compliance (reviewer still
  catches any hands-off violations because the gate is provider-
  agnostic).

If OpenRouter engineer meets criteria: opt-in per project, with a
clear note in `projects.yaml.example` that bulk-scaffolding work
routes OK and complex / algorithmic work should stay on Claude.

If it falls short: stay on Claude; revisit with paid-tier Qwen
Coder Plus or a different provider.

**Day 1 (implementation landing):**

**Day 7+ (steady state):**

  config-adjacent section.

## Open questions

- **Does aider's SEARCH/REPLACE protocol round-trip cleanly through
  git worktrees?** GS uses `.bot-worktree` per cycle; aider creates
  its own git commits by default. Need to `--no-auto-commits` and let
  GS handle commits externally.
- **How does aider handle the `engineer_command.sh ${cycle_budget_minutes}`
  wrapper contract?** Aider doesn't have a budget concept natively;
  might need a timeout + cancellation hook at the GS level.
- **What's the right granularity for model choice?** One model per
  project (current sketch), or per-task (e.g., hard tasks to Claude,
  easy ones to Qwen)? Per-project simpler; per-task more optimal.
  Start per-project.
- **Does Qwen3 Coder Plus handle TypeScript + React fluently?** The
  benchmark will answer this. Anecdotally yes, but worth measuring.

## Non-goals for Phase 7

- **NOT replacing Claude for hard tasks.** Claude Code stays the
  default and the recommended engineer for complex work. Phase 7
  is about bulk scaffolding.
- **NOT building a custom OpenRouter wrapper yet.** Aider first; if
  aider fits, stop there. Custom wrapper is Phase 8+ if needed.
- **NOT auto-routing tasks by complexity.** The operator picks the
  engineer per project in v1. Auto-routing is a future optimization.
- **NOT touching the reviewer.** Reviewer provider plumbing is
  already done and working. Phase 7 is strictly engineer-side.

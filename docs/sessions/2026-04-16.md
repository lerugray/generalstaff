### Dogfooding decision (commit 789699d)

Key decision: instead of using catalogdna as the first test
project, Ray proposed dogfooding — making GeneralStaff its own
first managed project. Rationale:

1. Avoids interfering with active catalogdna work
2. Exercises the greenfield path (new project, no legacy)
3. Self-referential test: if the dispatcher can improve its own
   codebase, that's a strong signal
4. Lower risk: no production consequences if cycles fail

This turned out to be the right call. The bot immediately had
real work to do (test coverage, code quality improvements) and
the feedback loop was tight — bugs in the dispatcher were
surfaced by running the dispatcher.

### Worktree isolation

Bot works in `.bot-worktree` on the `bot/work` branch. Ray
stays on `master`. No branch-switching conflicts. Merges from
bot/work into master happen between cycles. This is the same
pattern catalogdna uses, validated over 22+ bot runs there.

### Bug #1: branch-awareness (commit 26bbf2b)

First live cycle surfaced a real bug: `cycle.ts` captured SHAs
from HEAD (which was master, the interactive branch) instead of
from the bot/work branch where the engineer actually committed.
Result: the diff passed to the reviewer was always empty. The
reviewer correctly identified this as "nothing to verify" and
returned a weak/failed verdict.

This is a textbook "initial negative" — the system surfaced its
own bug on the very first run. Fixed by making cycle.ts
branch-aware: it now reads SHAs from bot/work explicitly.

### Bug #2: auto-commit of state artifacts (commits 0467d53, b089730)

Between cycles, the dispatcher auto-commits state artifacts
(fleet_state.json, cycle logs). Two issues:

1. First fix: auto-commit wasn't happening at all between cycles
2. Second fix: `git add` failed on gitignored paths. Switched
   to `--ignore-errors` flag for state artifact commits.

Both were plumbing issues, not architectural problems. Fixed
within the same session.

### Windows-specific findings

Worktree cleanup on Windows is tricky:
- Stale git worktree registrations persist after directory removal

**First observation run (90 min budget):** 3 cycles, all
"verified"... and **all three implemented gs-056** (log
rotation in `clean.ts`). Redundant work. Dig uncovered the
root cause: the dispatcher's pre-cycle "auto-update bot/work
to HEAD" step (`cycle.ts:295`, from gs-012) was **resetting
bot/work to master's HEAD before merging the previous cycle's
work in**. No merge logic existed anywhere in the dispatcher
— on the work PC the merges had been happening manually. On
a fresh home-PC session with zero manual merges, every
cycle's engineer commit got orphaned when the next cycle
started.

This is a **dispatcher-level stupid+industrious failure**.
The engineer was clever (each diff correctly implemented
gs-056 and verified). The reviewer was correct (each verdict
matched the diff). The *orchestrator* was stupid+industrious:
confidently running cycles end-to-end and throwing the output
away. A rich Hammerstein moment, captured in detail in
`docs/internal/Hammerstein Observations - Claude.md` (commit
`4d08623`).

**Fix + recovery:**

- `cycle.ts` now calls `countCommitsAhead(branch, HEAD)`
  before the reset. If >0: with `auto_merge: true`, merge
  `--no-ff` into HEAD first; with `auto_merge: false`, skip
  the cycle and print a clear error with exact remediation
  instructions. Commit `00e727f`.
- `projects.yaml` (local, gitignored) opted dogfood into
  `auto_merge: true`. We had 35+ clean verified cycles, well
  past Hard Rule #4's 5-cycle threshold.
- Manually merged `bot/work` into `master` to recover the one
  good copy of gs-056 (`319bb65`). Commit `7e3af65`.

**Verification run (90 min budget):** 3 cycles, all
verified, **three different tasks this time**: gs-057 (digest
command), gs-058 (per-cycle console log), gs-059 (disk
usage in summary). Auto-merging visible between cycles as
designed (commits `2b28d8f` and `8140749`). Fix validated
end-to-end.

**Session-end merge gap (follow-on):** the per-cycle
auto-merge fires at cycle *start*, so the final cycle of a
session has no "next cycle" to merge it in. Cycle 3's
gs-059 sat on bot/work after the verify run ended. Added a
session-end merge pass in `session.ts` that, for each
`auto_merge: true` project that ran ≥1 cycle, merges its
bot branch into HEAD before writing the digest. Updated
three test-helper mocks to include
`countCommitsAhead: async () => 0`. Commit `ac4bff8`.

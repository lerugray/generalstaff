## Key insight from the catalogdna deep-dive

catalogdna's bot is mature: git worktrees (`.bot-worktree`),
heartbeat-pushing during long runs (so Ray can monitor from
another machine), auto-publish via `run_bot_publish.sh`,
Phase A/B clever-lazy framework baked into `CLAUDE-AUTONOMOUS.md`,
sub-agent delegation rules, chrome-review file-based protocol
for webapp work, and a full pre-flight + wind-down protocol.

**GeneralStaff Phase 1 wraps this bot. It does not replace it.**

GeneralStaff adds three things catalogdna's bot doesn't have:

1. **Independent verification gate** — runs catalogdna's
   pytest+ruff externally, *separately from what the bot did
   internally*, to catch cases where the model claimed tests
   passed but they didn't (Polsia's #1 failure mode).
2. **Scope-match Reviewer** — a fresh `claude -p` with read+git
   tools that examines the diff and the marked-done tasks and
   confirms they actually match.
3. **Open audit log** — `PROGRESS.jsonl` with full prompts,
   responses, tool calls, diffs, and verification output for every
   cycle.

These are the three differentiators vs. Polsia. They work for
catalogdna because catalogdna's bot is already disciplined; they
will work for any project, even one without an existing bot.

## The two abstractions that keep this open-source-shaped

Per Ray's 2026-04-15 evening guidance, even though Phase 1 is a
prototype against catalogdna, the architecture has to make sense
as an open-source tool in general. Two fields in `projects.yaml`
do the work:

```yaml
projects:
  - id: catalogdna
    path: C:/Users/rweis/OneDrive/Documents/catalogdna
    engineer_command: "bash run_bot.sh ${cycle_budget_minutes}"
    verification_command: "bash scripts/verify_for_generalstaff.sh"
    cycle_budget_minutes: 60
    branch: bot/work
    auto_merge: false
    hands_off:
      - src/catalogdna/interpret/
      - CLAUDE.md
      - CLAUDE-AUTONOMOUS.md
      - .claude/
      - run_bot*.sh
      - run_bot*.bat
      - bot_tasks.md
      - scripts/bot_heartbeat.sh
      - scripts/chrome_review*.sh
      - scripts/worktree_venv.py
```

**`engineer_command`** is the abstraction that decouples
GeneralStaff from any specific bot invocation:

- Projects with mature bot infra (catalogdna) shell out to their
  existing launcher
- Greenfield projects use a default that GeneralStaff provides:

  ```
  default_engineer_command: |
    claude -p "Read .generalstaff/MISSION.md and
    .generalstaff/tasks.json, pick the highest-priority unfinished
    task, work on it on the bot/work branch, commit when tests
    pass." \
      --allowedTools "Read,Write,Edit,Bash,Grep,Glob"
  ```

**`verification_command`** is the abstraction that makes the
verification gate independent of whatever the engineer did:

- Runs after `engineer_command` exits, regardless of whether the
  engineer claimed tests passed
- For catalogdna: a thin script (`scripts/verify_for_generalstaff.sh`)
  that runs pytest with the right ignores plus ruff
- For greenfield projects: defaults to whatever the project's
  test runner is (`pytest`, `bun test`, `cargo test`, etc.)

These two fields plus `hands_off` and `branch` are all the
project-specific configuration the dispatcher needs. Everything
else is universal.

## Architecture: session loop with cycle chaining

```
generalstaff session [--budget=480]    # 8-hour overnight session

  while time_remaining > cycle_budget:
      if STOP exists: exit
      pick next project (priority * staleness, override file)
      if no project eligible: exit
      run one cycle on that project
        - engineer_command (with sub-budget = cycle_budget_minutes)
        - verification_command
        - reviewer agent (fresh claude -p)
        - write audit log entries
      update fleet state
      decide: chain another cycle on same project, or rotate?

  morning digest with all cycles
```

**Cycle = one bot run.** Within a session, multiple cycles happen.
A session can run multiple cycles on the same project if it has
more work, or rotate to other projects. This matches "harvest
unattended hours into reviewable progress" — overnight session,
many cycles, morning digest.

The picker decides between "chain on same project" vs. "rotate"
based on:
- Whether the previous cycle succeeded (don't keep grinding a
  project that just verification_failed)
- Whether the project has more work (e.g., `bot_tasks.md` still
  has unchecked items for catalogdna)
- Whether other projects are starved (priority × staleness)

For Phase 1 with only catalogdna registered, the picker always
returns catalogdna until it has no more work.

## Phase 1 cycle workflow (concrete, for catalogdna)

1. **Pre-flight in the dispatcher:**
   - Check `STOP` file at GeneralStaff fleet root
   - Check that catalogdna's working tree is clean (no uncommitted
     changes on master)
   - Check that catalogdna's `.bot-worktree/` does NOT exist (= no
     other run_bot.sh in flight)
   - If either check fails, log + skip catalogdna for this cycle
   - Capture `cycle_start_sha = git rev-parse HEAD` in catalogdna
2. **Skip Planner for catalogdna.** catalogdna's bot reads
   `bot_tasks.md` itself via its CLAUDE-AUTONOMOUS.md protocol.
   GeneralStaff's Planner agent is a no-op for projects that have
   their own task selection. (For greenfield projects in Phase 3+,
   Planner picks from `.generalstaff/tasks.json`.)
3. **Engineer step:** Invoke `engineer_command` =
   `bash run_bot.sh 60`. This is a long-running subprocess.
   Capture stdout/stderr to a per-cycle log file. Wait for clean
   exit or hard timeout (cycle_budget_minutes + 5 min grace).
4. **Capture cycle_end_sha** from catalogdna.
5. **Independent verification gate:** Run `verification_command`
   = `bash scripts/verify_for_generalstaff.sh`. This wraps:
   ```bash
   PYTHONPATH=src .venv/Scripts/python -m pytest tests/ -q \
     --ignore=tests/test_property.py --ignore=tests/test_tui.py
   PYTHONPATH=src .venv/Scripts/python -m ruff check src/ tests/
   ```
   Capture exit code and full output. Expected: tests pass count
   matches catalogdna's known baseline (currently 654 passed,
   3 skipped).
6. **Diff capture:** `git diff cycle_start_sha cycle_end_sha`
   in catalogdna. Save as `diff.patch`.
7. **Reviewer agent:** Spawn fresh `claude -p` with read-only +
   git tools, hand it:
   - The diff
   - The list of `[x]` items that appeared in `bot_tasks.md`
     during the cycle (parsed from before/after)
   - The cycle's session note from `docs/Sessions/` (if the bot
     wrote one)
   - The verification command output
   - The hands_off list
   
   Ask: *"Does this diff match the marked-done tasks? Any scope
   drift? Any hands_off violations? Verdict: verified /
   verified_weak / verification_failed, with reason."*
   
   Reviewer returns a structured verdict (JSON).
8. **Three-state outcome:**
   - **`verified`** — verification command exited 0 AND reviewer
     said scope matches → ready_to_merge in digest (auto_merge
     stays OFF for catalogdna's first 5 GeneralStaff cycles)
   - **`verified_weak`** — verification passed but reviewer said
     "looks right but I can't fully confirm" (rare for
     catalogdna; common for projects with weak verification
     commands) → escalate to digest
   - **`verification_failed`** — verification command failed, OR
     reviewer found scope drift, OR reviewer found hands_off
     violation → escalate to digest, branch preserved for human
     review
9. **Audit log writes (Hard Rule #9):** PROGRESS.jsonl gets
   entries for every step:
   - `cycle_start` — timestamp, project, cycle_id, start_sha
   - `engineer_invoked` — full command, env vars, working dir
   - `engineer_completed` — exit code, duration, stdout/stderr
     truncated to ~10KB head/tail
   - `verification_run` — full command + full output
   - `verification_outcome` — verified | failed
   - `diff_summary` — stats from `git diff --stat`
   - `reviewer_invoked` — full prompt sent to claude -p
   - `reviewer_response` — full response from claude -p
   - `reviewer_verdict` — structured verdict
   - `cycle_end` — timestamp, end_sha, final state
   
   Append-only. Truncation note: full engineer stdout/stderr is
   saved separately to `cycles/${cycle_id}/engineer.log`; the
   PROGRESS.jsonl entry just references the file.
10. **Cycle chaining decision:**
    - Time remaining in session > `cycle_budget_minutes + 5 min`?
    - catalogdna's `bot_tasks.md` still has unchecked `[ ]` items?
    - Last cycle was NOT verification_failed?
    
    All three yes → schedule another catalogdna cycle. Otherwise
    → rotate (Phase 1 has only catalogdna so → exit session).
11. **Update `fleet_state.json`:** cycle count, last cycle outcome,
    last cycle time, accumulated time per project.

## Phase 1 file structure (revised — smaller)

```
GeneralStaff/
├── src/
│   ├── cli.ts                # commands: session, cycle, status, stop, log
│   ├── session.ts            # session loop with cycle chaining
│   ├── dispatcher.ts         # picker (priority * staleness + chaining rules)
│   ├── cycle.ts              # one cycle: engineer + verify + review + log
│   ├── engineer.ts           # subprocess wrapper for engineer_command
│   ├── verification.ts       # subprocess wrapper for verification_command
│   ├── reviewer.ts           # spawn claude -p reviewer, parse verdict
│   ├── audit.ts              # PROGRESS.jsonl append-only writer
│   ├── state.ts              # STATE/HANDOFF/fleet_state read+write
│   ├── projects.ts           # projects.yaml parser + validator
│   ├── safety.ts             # STOP file checker, hands_off validation
│   ├── digest.ts             # local digest writer (Phase 2: + Telegram)
│   └── types.ts              # shared TS types for state files
├── tests/
│   ├── state.test.ts
│   ├── projects.test.ts
│   ├── audit.test.ts
│   ├── verification.test.ts
│   └── dispatcher.test.ts
├── package.json
├── tsconfig.json
├── bunfig.toml
└── (existing GeneralStaff design docs)
```

Notice: **no `agents/` subdirectory.** Engineer is shelled out
(not a `claude -p` invocation), so only the Reviewer is a `claude
-p` call. Two subprocess wrappers (`engineer.ts`,
`verification.ts`) plus one Reviewer module (`reviewer.ts`) is
the entire agent surface for Phase 1.

## Build order

1. **Project bootstrap.** `package.json`, `tsconfig.json`,
   `bunfig.toml`, basic CLI scaffolding. `bun --version` works,
   `bun run cli.ts --version` returns "0.0.1".
2. **Type definitions.** `src/types.ts` — STATE, HANDOFF, fleet
   state, projects.yaml schema, PROGRESS.jsonl entry types,
   reviewer verdict enum. Lock the JSON contracts before logic.
3. **State module.** `src/state.ts` — atomic file writes (write
   to tmp + rename), per-project state read/write helpers.
4. **Projects loader.** `src/projects.ts` — parse YAML, validate:
   - `hands_off` must be non-empty
   - `engineer_command` must be non-empty
   - `verification_command` must be non-empty (warn + flag
     `verified_weak` if it's `true` / `:` / empty)
   - `cycle_budget_minutes` must be a positive integer
5. **Safety module.** `src/safety.ts` — STOP file checker, working
   tree clean check, `.bot-worktree` existence check (catalogdna-
   specific safety to avoid clashing with manual run_bot.sh).
6. **Audit writer.** `src/audit.ts` — append-only PROGRESS.jsonl
   with the schema from `DESIGN.md` v2 + the entry types listed
   in §9 above. Atomic appends.
7. **Engineer module.** `src/engineer.ts` — subprocess wrapper
   for `engineer_command`. Streaming stdout/stderr capture.
   Hard timeout. Returns structured result.
8. **Verification module.** `src/verification.ts` — subprocess
   wrapper for `verification_command`. Captures exit code and
   full output. Returns `verified` / `failed` / `weak`.
9. **Reviewer module.** `src/reviewer.ts` — spawn `claude -p`
   with `--allowedTools "Read,Bash,Grep,Glob"` (read+git only,
   no Write/Edit). Structured prompt template. Parse JSON
   verdict from response. Handle malformed responses gracefully
   (= `verification_failed`).
10. **Cycle module.** `src/cycle.ts` — orchestrate engineer +
    verification + reviewer + audit log. Capture cycle_start_sha
    and cycle_end_sha. Build the audit log entries.
11. **Dispatcher / picker.** `src/dispatcher.ts` — priority ×
    staleness picker, chaining rules (same project vs. rotate),
    `next_project.txt` override file support.
12. **Session module.** `src/session.ts` — outer loop, time
    budget management, calls dispatcher → cycle → repeat.
13. **CLI entry point.** `src/cli.ts` — wire commands:
    - `generalstaff session [--budget=N]` — run a session
    - `generalstaff cycle --project=ID` — run one cycle
    - `generalstaff status` — show fleet state
    - `generalstaff stop` / `generalstaff start` — STOP file
    - `generalstaff log [--project=ID]` — tail PROGRESS.jsonl
14. **catalogdna integration:**
    - Create `.generalstaff/MISSION.md` in catalogdna based on
      the existing CLAUDE-AUTONOMOUS.md
    - Write `scripts/verify_for_generalstaff.sh` (the wrapper
      script for the verification command)
    - Write a real `projects.yaml` in GeneralStaff with the
      catalogdna entry from §the abstractions above
    - Add `.generalstaff/` to catalogdna's `.gitignore` initially
      (so test runs don't pollute catalogdna's git history before
      the design stabilizes)
15. **Dry-run on catalogdna:** `--dry-run` flag captures
    catalogdna state and *simulates* engineer/verify/review
    without actually running them. Validates the dispatcher
    plumbing without touching catalogdna.
16. **First live cycle (supervised):** Manual trigger. Ray is
    NOT actively coding catalogdna. Auto-merge OFF. Review the
    PROGRESS.jsonl entries and the local digest by hand.
17. **Five clean cycles:** Validate the verification gate over
    a week of supervised runs (one cycle per night, not five
    cycles in one night) before considering Phase 1 done. Each
    cycle is reviewed by hand; Ray reviews the digest in the
    morning.
18. **Cycle chaining test:** Once individual cycles are stable,
    run `generalstaff session --budget=180` and verify it runs
    2-3 catalogdna cycles in sequence within the session.

## Definition of done for Phase 1

- `generalstaff session --budget=60` runs end-to-end on catalogdna
  during a non-Ray-coding window
- A test failure in catalogdna correctly produces
  `verification_failed` that does not get marked done
- A scope-drift case (Engineer touches a file outside the task
  scope) is caught by the Reviewer and produces
  `verification_failed`
- A hands_off violation case (Engineer modifies a file in the
  hands_off list) is caught by the Reviewer and produces
  `verification_failed`
- `PROGRESS.jsonl` contains a complete audit trail of one cycle
  with all event types
- Local digest file is written with verdict + summary
- Five supervised cycles run cleanly (one per night, manual
  review of each)
- Cycle chaining: a single session runs at least 2 cycles on
  catalogdna in sequence

After all of these, Phase 2 (formal Reviewer pass with
permission-level hands_off enforcement, plus second project
candidate) can begin.

## What Phase 1 explicitly does NOT include

- **Permission-level hands_off enforcement.** catalogdna does it
  via instructions; we inherit the same approach. Phase 2 or 3
  upgrades to Claude Code permission deny rules.
- **Default `engineer_command` for greenfield projects.** Phase 1
  only tests the wrap path. The default is designed in Phase 3
  when we add Retrogaze or Sandkasten.
- **Local UI.** Phase 4-5.
- **Telegram digest.** Phase 2 — for Phase 1, local digest files
  only.
- **Auto-merge.** OFF by default, requires 5 clean cycles per
  project, and even then it's opt-in per `auto_merge: true` in
  projects.yaml.
- **Plugin API for creative roles.** Phase 10+.
- **Cron / Task Scheduler integration.** Phase 1 is manual
  triggers and supervised runs. Once stable, a `.bat` wrapper +
  Task Scheduler is Phase 1.5.

## Key insight from the catalogdna deep-dive

catalogdna's bot is mature: git worktrees (`.bot-worktree`),
heartbeat-pushing during long runs (so Ray can monitor from
another machine), auto-publish via `run_bot_publish.sh`,
Phase A/B clever-lazy framework baked into `CLAUDE-AUTONOMOUS.md`,
sub-agent delegation rules, chrome-review file-based protocol
for webapp work, and a full pre-flight + wind-down protocol.

**GeneralStaff Phase 1 wraps this bot. It does not replace it.**

GeneralStaff adds three things catalogdna's bot doesn't have:

1. **Independent verification gate** — runs catalogdna's
   pytest+ruff externally, *separately from what the bot did
   internally*, to catch cases where the model claimed tests
   passed but they didn't (Polsia's #1 failure mode).
2. **Scope-match Reviewer** — a fresh `claude -p` with read+git
   tools that examines the diff and the marked-done tasks and
   confirms they actually match.
3. **Open audit log** — `PROGRESS.jsonl` with full prompts,
   responses, tool calls, diffs, and verification output for
   every cycle, living in GeneralStaff's repo.

These are the three differentiators vs. Polsia. They work for
catalogdna because catalogdna's bot is already disciplined; they
will work for any project, even one without an existing bot.

## Safety architecture: state lives in GeneralStaff, not in catalogdna

Per `PHASE-1-RESOLUTIONS` §Q5, GeneralStaff writes **nothing**
to catalogdna's working tree. The breakdown:

- GeneralStaff reads catalogdna freely (bot_tasks.md,
  bot_status.md, git state, docs/Sessions/)
- GeneralStaff writes only to `state/${project_id}/` inside
  GeneralStaff's own git repo
- The Engineer subprocess (`bash run_bot.sh`) runs with
  `cwd=catalogdna` and writes to catalogdna's working tree on
  the `bot/work` branch — but that's catalogdna's own bot
  doing its own thing, unchanged from current behavior
- The Reviewer agent runs with `cwd=catalogdna` (so it can
  `git log`, `git diff`, read catalogdna files) but its
  prompt, response, and verdict are written to
  `state/catalogdna/cycles/${cycle_id}/` by GeneralStaff, not
  to catalogdna

This eliminates cross-project contamination risk at the source.

## The two abstractions that keep this open-source-shaped

Per Ray's 2026-04-15 evening guidance, even though Phase 1 is a
prototype against catalogdna, the architecture has to make sense
as an open-source tool in general. Fields in `projects.yaml` do
the work:

```yaml
projects:
  - id: catalogdna
    path: C:/Users/rweis/OneDrive/Documents/catalogdna
    engineer_command: "bash run_bot.sh ${cycle_budget_minutes}"
    verification_command: >-
      PYTHONPATH=src .venv/Scripts/python -m pytest tests/ -q
      --ignore=tests/test_property.py --ignore=tests/test_tui.py
      && PYTHONPATH=src .venv/Scripts/python -m ruff check src/ tests/
    cycle_budget_minutes: 60
    work_detection: catalogdna_bot_tasks
    concurrency_detection: catalogdna
    branch: bot/work
    auto_merge: false
    hands_off:
      - src/catalogdna/interpret/
      - CLAUDE.md
      - CLAUDE-AUTONOMOUS.md
      - .claude/
      - run_bot*.sh
      - run_bot*.bat
      - bot_tasks.md
      - scripts/bot_heartbeat.sh
      - scripts/chrome_review*.sh
      - scripts/worktree_venv.py

dispatcher:
  state_dir: ./state
  fleet_state_file: ./fleet_state.json
  stop_file: ./STOP
  override_file: ./next_project.txt
  picker: priority_x_staleness
  max_cycles_per_project_per_session: 3
```

**`engineer_command`** is the abstraction that decouples
GeneralStaff from any specific bot invocation:

- Projects with mature bot infra (catalogdna) shell out to
  their existing launcher
- Greenfield projects use a default that GeneralStaff provides
  (designed in Phase 3):

  ```
  default_engineer_command: |
    claude -p "Read MISSION.md and tasks.json, pick the
    highest-priority unfinished task, work on it on the
    bot/work branch, commit when tests pass." \
      --allowedTools "Read,Write,Edit,Bash,Grep,Glob"
  ```

**`verification_command`** is the abstraction that makes the
verification gate independent of whatever the engineer did:

- Runs after `engineer_command` exits, regardless of whether
  the engineer claimed tests passed
- For catalogdna: inline pytest + ruff (see YAML above)
- For greenfield projects: defaults to whatever the project's
  test runner is (`pytest`, `bun test`, `cargo test`, etc.)
- Intentionally **inline in projects.yaml**, NOT a wrapper
  script written into the managed project (that would be
  contamination)

**Two additional fields added in Phase 1 resolutions:**

- `work_detection`: how the dispatcher detects "this project
  has more work" for the chaining decision. Values:
  `catalogdna_bot_tasks` (parse catalogdna's bot_tasks.md for
  unchecked P0-P3 items) or `tasks_json` (parse
  `state/${id}/tasks.json`). Default: `tasks_json`.
- `concurrency_detection`: how the dispatcher detects "this
  project's bot is already running" before starting a cycle.
  Values: `catalogdna` (check `.bot-worktree`, `bot_status.md`,
  heartbeat sentinels) or `none`. Default: `none`.

Full detection logic in `PHASE-1-RESOLUTIONS-2026-04-15.md` §Q1
and §Q3.

## Architecture: session loop with cycle chaining

```
generalstaff session [--budget=480]    # 8-hour overnight session

  while time_remaining > cycle_budget:
      if STOP exists: exit
      pick next project (priority * staleness, override file)
      if no project eligible: exit
      check concurrency: isBotRunning(project)?
        yes → skip, log cycle_skipped, pick next project
        no  → proceed
      run one cycle on that project
        - engineer_command (sub-budget = cycle_budget_minutes)
        - verification_command
        - reviewer agent (fresh claude -p)
        - write audit log entries to state/${id}/
      update fleet state
      decide: shouldChain() → chain same project OR rotate?

  morning digest with all cycles
```

**Cycle = one bot run.** Within a session, multiple cycles
happen. A session can run multiple cycles on the same project
if it has more work, or rotate to other projects. This matches
"harvest unattended hours into reviewable progress" — overnight
session, many cycles, morning digest.

The picker decides "chain on same project" vs. "rotate" using
the rules from `PHASE-1-RESOLUTIONS` §Q1: last cycle NOT
`verification_failed`, per-project cycle cap not reached,
sufficient budget remaining, `hasMoreWork(project)` returns
true.

For Phase 1 with only catalogdna registered, the picker always
returns catalogdna until it has no more work or the cap is
reached.

## Phase 1 cycle workflow (concrete, for catalogdna)

1. **Pre-flight in the dispatcher:**
   - Check `STOP` file at GeneralStaff fleet root
   - Check catalogdna's working tree is clean (no uncommitted
     changes on master)
   - **Concurrency check** (Q3): `isBotRunning(catalogdna)`
     returns false across all three signals. If it returns
     true, skip with a `cycle_skipped` audit entry.
   - Capture `cycle_start_sha = git rev-parse HEAD` in
     catalogdna

2. **Skip Planner for catalogdna.** catalogdna's bot reads
   `bot_tasks.md` itself via its CLAUDE-AUTONOMOUS.md protocol.
   GeneralStaff's Planner agent is a no-op for projects whose
   `work_detection` is `catalogdna_bot_tasks`. (For greenfield
   projects in Phase 3+, Planner picks from `tasks.json`.)

3. **Engineer step:** Invoke `engineer_command` =
   `bash run_bot.sh 60` with `cwd=catalogdna`. Long-running
   subprocess. Capture stdout/stderr to
   `state/catalogdna/cycles/${cycle_id}/engineer.log`. Wait for
   clean exit or hard timeout (cycle_budget_minutes + 5 min
   grace).

4. **Capture cycle_end_sha** from catalogdna.

5. **Independent verification gate:** Run `verification_command`
   (the inline pytest + ruff command from projects.yaml) with
   `cwd=catalogdna`. Capture exit code and full output to
   `state/catalogdna/cycles/${cycle_id}/verification.log`.
   Expected baseline: 654 passed, 3 skipped (may drift as
   catalogdna evolves).

6. **Diff capture:** `git diff cycle_start_sha cycle_end_sha`
   in catalogdna. Save to
   `state/catalogdna/cycles/${cycle_id}/diff.patch`.

7. **Reviewer agent:** Spawn fresh `claude -p` with
   `--allowedTools "Read,Bash,Grep,Glob"` and `cwd=catalogdna`.
   Fill the Reviewer prompt template (Q2) with:
   - The full diff
   - The list of `[x]` items newly appearing in `bot_tasks.md`
     during the cycle
   - The cycle's session note from `docs/Sessions/` (if
     written)
   - The verification command output
   - The hands_off list
   
   Write the filled prompt to
   `state/catalogdna/cycles/${cycle_id}/reviewer-prompt.txt`,
   spawn, write the response to
   `state/catalogdna/cycles/${cycle_id}/reviewer-response.txt`.
   Parse JSON verdict; malformed → default to
   `verification_failed`.

8. **Three-state outcome:**
   - **`verified`** — verification exited 0 AND reviewer said
     scope matches → ready_to_merge in digest (auto_merge OFF
     until 5 clean cycles per project)
   - **`verified_weak`** — verification passed but reviewer
     said "looks right but can't fully confirm" (rare for
     catalogdna) → escalate to digest
   - **`verification_failed`** — verification exit non-zero,
     OR reviewer found scope drift, OR hands_off violation, OR
     malformed reviewer response → escalate to digest, branch
     preserved for human review

9. **Audit log writes (Hard Rule #9):**
   `state/catalogdna/PROGRESS.jsonl` gets entries for every
   step (cycle_start, engineer_invoked, engineer_completed,
   verification_run, verification_outcome, diff_summary,
   reviewer_invoked, reviewer_response, reviewer_verdict,
   cycle_end). Append-only. Large artifacts (engineer stdout,
   reviewer prompt/response) live in per-cycle files
   referenced from the PROGRESS entries, not inlined.

10. **Cycle chaining decision (Q1 logic):**
    - `lastCycle.verdict !== "verification_failed"` AND
    - `cyclesOnThisProject < max_cycles_per_project_per_session`
      (default 3) AND
    - `time_remaining > cycle_budget_minutes + 5` AND
    - `hasMoreWork(catalogdna)` returns true
    
    All yes → schedule another catalogdna cycle. Otherwise →
    rotate (Phase 1 has only catalogdna → exit session).

11. **Update `fleet_state.json`:** cycle count, last outcome,
    last cycle time, accumulated time per project.

**Two properties worth calling out:**

1. Nothing under `state/` or anywhere else in GeneralStaff points
   into a managed project's working tree. All state files live
   in GeneralStaff's own repo. The dispatcher reads from managed
   projects via absolute paths and writes only under `state/`.
   This is the Q5 safety decision.

2. No `src/agents/` subdirectory. Engineer is shelled out (not a
   `claude -p` invocation), so only the Reviewer is a `claude
   -p` call. Two subprocess wrappers (`engineer.ts`,
   `verification.ts`) plus one Reviewer module (`reviewer.ts`)
   is the entire agent surface for Phase 1.

## Definition of done for Phase 1

- `generalstaff session --budget=60` runs end-to-end on catalogdna
  during a non-Ray-coding window
- A test failure in catalogdna correctly produces
  `verification_failed` that does not get marked done
- A scope-drift case (Engineer touches a file outside the task
  scope) is caught by the Reviewer and produces
  `verification_failed`
- A hands_off violation case (Engineer modifies a file in the
  hands_off list) is caught by the Reviewer and produces
  `verification_failed`
- `state/catalogdna/PROGRESS.jsonl` contains a complete audit
  trail of one cycle with all event types
- Local digest file written with verdict + summary
- Five supervised cycles run cleanly (one per night, manual
  review of each)
- Cycle chaining: a single session runs ≥2 cycles on catalogdna
  in sequence
- **Zero writes to catalogdna's working tree** verified after
  each cycle with `cd catalogdna && git status --porcelain`
  (bot/work branch commits are catalogdna's bot's own writes,
  not GeneralStaff's)

After all of these, Phase 2 (formal Reviewer pass with
permission-level hands_off enforcement, plus second project
candidate) can begin.

## What Phase 1 explicitly does NOT include

- **Permission-level hands_off enforcement.** catalogdna does it
  via instructions; we inherit the same approach. Phase 2 or 3
  upgrades to Claude Code permission deny rules.
- **Default `engineer_command` for greenfield projects.** Phase 1
  only tests the wrap path. The default is designed in Phase 3
  when we add Retrogaze or Sandkasten.
- **Local UI.** Phase 4-5.
- **Telegram digest.** Phase 2 — for Phase 1, local digest files
  only.
- **Auto-merge.** OFF by default, requires 5 clean cycles per
  project, and even then it's opt-in per `auto_merge: true` in
  projects.yaml.
- **Plugin API for creative roles.** Phase 10+.
- **Cron / Task Scheduler integration.** Phase 1 is manual
  triggers and supervised runs. Once stable, a `.bat` wrapper +
  Task Scheduler is Phase 1.5.

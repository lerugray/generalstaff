# Pivot — 2026-04-15

**From:** Personal nightly meta-dispatcher for Ray's local projects
**To:** Open-source autonomous engineering platform — the principled
alternative to Polsia

**Decided:** 2026-04-15 interactive session
**Authority:** Ray, after a deep-dive research pass on Polsia
(polsia.com, Trustpilot, Ben Broca's Indie Hackers writeup, Mixergy
interview, Context Studios analysis, PolsiaAI GitHub org)

---

## The old mission (verbatim from README.md)

> "Cross-project autonomous bot dispatcher. Runs Claude Code agents
> on multiple of Ray's local projects in sequence, overnight, while
> Ray sleeps."

The original framing was personal infra: Ray is the only user, the
goal is to harvest unattended hours into reviewable progress across
Ray's project portfolio. catalogdna would be the cornerstone, with
1-2 other Ray projects added later. Distribution = a private folder.

## The new mission

> "Open-source autonomous engineering for solo founders.
> Your code, your keys, your control.
> The principled alternative to closed-source bot platforms."

GeneralStaff is now a public product. Other people will run it. The
audience is solo engineers and technical founders who want
autonomous engineering work done on their own projects without
giving up code ownership, paying revenue shares, or trusting an
opaque SaaS platform with their git history.

## Why pivot

The deep-dive on Polsia revealed that the deepest user complaints
map almost 1:1 onto safeguards already designed in `DESIGN.md` and
already battle-tested in catalogdna's Phase A/B verification
protocol. **GeneralStaff is ~70% of an open-source Polsia
alternative as currently designed.** The pivot mostly inherits and
formalizes work that already exists.

The complaints that map directly onto existing GeneralStaff design:

| Polsia complaint | GeneralStaff design element |
|---|---|
| "AI marks tasks complete without verifying" | Verification ritual (research-notes.md synthesis #2) |
| "Can't export my code" | Per-project `bot/work` branch on user's own git (DESIGN.md ll. 102-123) |
| "AI made commitments I can't undo" | `STOP` kill switch + hands-off lists (safety rules #5, #11) |
| "Charged credits for failed work" | BYOK model — no credit system to weaponize |
| "Generic marketing slop" | Existing Hard Rule #1 (no creative delegation) |
| "Bot wrecked things outside its lane" | Hands-off lists at tool permission level (safety rule #9) |
| "No idea what my bot is doing" | PROGRESS.jsonl audit log (extended in v2) |

The remaining ~30% is: a local UI shell, BYOK config plumbing,
public documentation, an installable distribution, and the formal
verification gate (currently described as a "ritual," now becoming
a load-bearing rule per RULE-RELAXATION-2026-04-15.md §3).

## Strategic positioning

> "GeneralStaff: open-source autonomous engineering for solo founders.
>  Your code stays in your git. Your keys stay in your shell.
>  Your bot only marks work done after your tests pass."

The Hammerstein framing is the moat. Polsia ships agents in all
four quadrants of Hammerstein's officer typology — including the
**stupid + industrious** quadrant, where the agent is confidently
producing slop (false task completions, generic marketing copy,
fake data pulls, auto-committed influencer giveaways).

Hammerstein's actual position was that stupid+industrious officers
must be **dismissed at once** because they cause unbounded damage.
Ray summarized this in the 2026-04-15 session: *"the service isn't
useful if all the employees are stupid and industrious."*

GeneralStaff's commitment is to keep agents in the
**clever + industrious** quadrant ("general staff" — execution at
the right scope, where industriousness compounds positively) and
the **clever + lazy** quadrant (Reviewer/Planner roles — judgment
without overactivity). The stupid+industrious quadrant is
structurally prevented by:

1. The verification gate (Hard Rule #6) — bots cannot mark fake
   work done.
2. The hands-off lists (Hard Rule #5) — bots cannot wreck things
   outside their lane.
3. The default-off creative roles (Hard Rule #1) — bots are not
   asked to make taste judgments they will get wrong.
4. The open audit log (Hard Rule #9) — every action is visible
   before it compounds.
5. The user's own git remote (Hard Rule #7) — a misbehaving
   branch can be deleted in one command.

This is not a marketing claim — it is the architecture of the
product, and it is the entire reason the open-source version can
beat a funded SaaS competitor on something other than distribution.

The catalogdna `AI Collaboration Principles.md` document is the
canonical writeup of the underlying philosophy. The open-source
README will reference it as the foundational text.

## What changes from the original GeneralStaff design

| Aspect | Old (personal infra) | New (open-source product) |
|---|---|---|
| Audience | Ray | Solo engineers / technical founders |
| Distribution | Private folder | GitHub Releases + Homebrew + winget + npx |
| UI | None (file-based only) | Local desktop UI (Tauri) over file state |
| LLM auth | Ray's `claude` CLI | BYOK (`ANTHROPIC_API_KEY` default) |
| Hosting | Ray's machine | User's machine (no GeneralStaff hosting) |
| Verification | Phase A/B "ritual" | Hard Rule #6 (load-bearing gate) |
| Audit log | Implicit (catalogdna logs) | Hard Rule #9 (full prompts/responses/diffs in PROGRESS.jsonl) |
| Code ownership | Ray's git | Hard Rule #7 (user's git, period) |
| Creative agents | Out of scope | Out of scope by default; opt-in plugins with explicit warnings |
| Hard rules | 5 | 10 (5 original + 5 new differentiators) |

## What stays from the original GeneralStaff design

- The Hammerstein philosophy and the GeneralStaff name
- The Planner → Engineer → Reviewer chain (now with verification gate)
- File-based per-project state (`MISSION.md`, `STATE.json`,
  `HANDOFF.md`, `tasks.json`, `PROGRESS.jsonl`, `REVIEW.md`)
- The dispatcher cycle (priority × staleness picker)
- Per-project hands-off lists (mandatory)
- The `STOP` kill-switch file
- Sequential MVP, parallel later
- Auto-merge OFF by default
- Cross-project write isolation (Engineer for project X cannot
  touch project Y)
- catalogdna remains the cornerstone reference implementation,
  unchanged

The pivot is **additive**, not destructive. Nothing in `DESIGN.md`'s
v1 section is being removed. The v2 section appended on 2026-04-15
extends it.

## Phased build plan (revised)

The original phased plan in `DESIGN.md` had 5 phases. The pivot
inserts new phases and renumbers. Updated plan:

- **Phase 0** (this conversation, 2026-04-15) — Pivot design docs.
  RULE-RELAXATION, PIVOT, DESIGN v2 section, README rewrite,
  CLAUDE.md update. **No code yet.**
- **Phase 1** — Sequential MVP for catalogdna with verification
  gate. Validates the dispatcher + verification gate end-to-end on
  Ray's known-good project before any external user touches it.
- **Phase 2** — Add Reviewer pass with the formal verification
  gate (was Phase 3 in original plan; now critical, no longer
  optional).
- **Phase 3** — Add second Ray project (Retrogaze experimental
  pipeline work or Sandkasten non-game-logic work). Validates the
  multi-project dispatcher under real load.
- **Phase 4** — Local UI shell (read-only views first). Tauri +
  read-only views of the file state. Lets you watch the bot work
  without intervening yet.
- **Phase 5** — Local UI control surface. Pause/resume, reorder
  priorities, intervene mid-cycle, approve/reject reviewer
  verdicts, view audit log searchably.
- **Phase 6** — Package as installable CLI + UI binary. GitHub
  Release, Homebrew tap, winget manifest, npx init.
- **Phase 7** — Public release on GitHub. Announce on HN,
  IndieHackers, /r/selfhosted, /r/LocalLLaMA. **No marketing-agent
  delegation** — Ray writes the launch posts himself, by hand,
  because the launch is exactly the kind of high-stakes creative
  work Hard Rule #1 protects from delegation.
- **Phase 8** — Parallel worktrees (was Phase 4 in original plan).
  Only after 2 weeks of clean sequential operation in production.
- **Phase 9** — SessionStart/SessionEnd hooks for state I/O (was
  Phase 5 in original plan).
- **Phase 10** — Optional creative role plugins, gated behind
  warnings.
- **Phase 11** — Optional self-hosted cloud mode (same code, run
  on a user-owned VPS). Never SaaS.

The catalogdna cornerstone reference is unchanged — its bot infra
is separate from GeneralStaff and continues to evolve in its own
repo. GeneralStaff wraps it as one of N projects, exactly as
originally designed.

## Polsia research summary

Founder Ben Broca, launched Dec 15 2025. $49/mo + 20% revenue
share. 1,000+ "companies" running on the platform. Reported
~$500k MRR within 3 months. Trustpilot 2.5/5 from 14 reviews
(65% one-star).

**Tech stack** (Indie Hackers founder disclosure): Node.js +
BullMQ + Redis worker queues, Render web hosting, Neon Postgres,
Postmark email, Stripe, Meta APIs. Claude Opus 4.6 as the
"primary reasoning model" for strategy. "Multi-layer memory
system" (undisclosed implementation). MCP integrations for
GitHub/email/Meta Ads/Stripe per the GitHub org README.

**Agent set:** CEO (strategy), Engineering (code/repo), Marketing
(ads/social), Support (customer email), Growth (metrics).

**What's missing under the hood:** No verification layer disclosed.
No human-in-the-loop gate beyond ad hoc. No documented guardrails.
The founder's own work-checking method is asking the model "are
you sure? Can you double check?" The founder admits a broken
support email route caused 20 unanswered Stripe disputes that
almost got their account flagged — they don't reliably monitor
their own infra. PolsiaAI GitHub org has one visible Python repo
with 3 stars; the platform is not open source in any meaningful
sense.

**Top user complaints** (Trustpilot, ranked):
1. False "task complete" markings — bots burn credits doing fake
   work
2. Code lock-in — no export, no zip, no GitHub access despite
   payment
3. Credits charged for failed work
4. Unresponsive support (week+ wait times for some users)
5. Auto-commitments the user can't undo
6. Generic marketing output (founder admitted)
7. Account suspension issues with no path to reinstatement

Every complaint maps to a specific Hard Rule above. Polsia's
weaknesses are GeneralStaff's product-market fit.

## Why this is winnable

Polsia won the launch news cycle and has marketing momentum that
a solo open-source competitor will not match on first release.
But:

1. **Polsia's tech moat is shallow.** Node + BullMQ + Redis +
   Claude API + ad hoc integrations. No verification layer, no
   real architecture, no open-source surface. The founder's own
   Indie Hackers post reads more like a distribution playbook
   than an engineering one.

2. **Their user complaints are structural, not bugs.** The
   complaints don't get fixed by Polsia shipping a patch — they
   get fixed by re-architecting around verification, code
   ownership, and BYOK, which is exactly what an open-source
   competitor can do from day one and Polsia probably can't do
   without breaking their revenue model (the 20% share depends
   on running the user's Stripe).

3. **The "principled alternative" niche is real and underserved.**
   HN, /r/selfhosted, /r/LocalLLaMA, the BYOK / self-hosted
   audience already exists and is hostile to revenue-share SaaS.
   They will adopt a credible open-source competitor.

4. **The Hammerstein framing is unique.** Nobody else is
   articulating *why* delegated creative work fails — they're
   either claiming it doesn't (Polsia) or refusing to delegate
   anything (manual development). The clever-vs-stupid /
   industrious-vs-lazy quadrant grid is a memorable, defensible
   philosophy.

5. **Vibecoder profile boost.** Ray shipping a credible
   open-source competitor to a YC-track SaaS in the autonomous-
   agents space is a portfolio piece that opens doors.

## Honest risks (carry these forward)

1. **Solo open-source vs. funded incumbent.** Distribution is the
   hard part. Plan for a slow-burn audience build, not a hockey
   stick.
2. **Anthropic ToS friction.** "Use your own subscription" is the
   strongest user benefit but the riskiest legal claim. API-key
   BYOK is unambiguously fine; subscription quota is murky and
   needs ToS verification before any public messaging.
3. **Maintenance burden.** Public open source means issues, PRs,
   security disclosures. Real ongoing work, not a write-once
   project.
4. **The marketing-rule trade-off cuts the addressable market.**
   Excluding creative agents on principle means the audience is
   "engineers who want autonomous engineering" not "founders who
   want autonomous companies." Smaller market, but defensible.
5. **Scope creep.** The Polsia-clone framing makes it tempting
   to ship features Polsia ships. Resist this. The MVP is:
   GeneralStaff dispatcher + verification gate + local UI
   viewer. That is it. Provisioning, multi-tenant, web SaaS,
   marketing agents — all explicitly Phase 10+ or never.

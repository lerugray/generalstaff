# UI Vision — 2026-04-19 (information architecture)

**Status:** Captured during the 2026-04-19 morning session as a
companion to `UI-VISION-2026-04-15.md` (which covered aesthetic /
theme). This doc is about **what the dashboard shows and when** —
the information architecture that determines which signals reach
the commander on which screens.

**Relationship to UI-VISION-2026-04-15.md:** the older doc
described *how* the UI looks (Kriegspiel / lithograph-map theme,
Phase 5.5+). This one describes *what* the UI shows and how it's
organized (Phase 6 onward). Both are forward-looking. The older
doc defers theming until mechanics are stable; this doc describes
the mechanics.

## Anchor: the dashboard is a subordinate's briefing

The mental model (Ray, 2026-04-19 morning): *"similar to an
overall report from a subordinate — projects the bot is working
on or has worked on, any successes/failures, project progress,
any projects that need tasks, and then options to do some of
those things. The dashboard should feel like a command center
where the user can get a bird's eye view on everything going on,
start dispatching orders/tasks and be able to get a picture of
how things are going relative to project priority, usage."*

This framing does a lot of work:

1. **It inherits Kriegspiel ops-room metaphors organically.** The
   commander receives a briefing; the staff officer (the bot)
   summarizes what happened. The dashboard is the briefing form,
   not a data explorer.
2. **It prioritizes by commander's-decision-urgency, not data
   category.** Things that need Ray's decision bubble up. Things
   that are running cleanly stay in the background.
3. **It sets the tone for copy.** "Your bot is working on gs-255"
   beats "Cycle 202604191425 in progress." Every label should
   read like a subordinate talking to their commander — concise,
   informative, action-oriented.
4. **It keeps non-programmers as the primary audience.**
   Programmers will use the CLI. The UI exists because a music
   teacher, a game designer, a small-business owner needs to
   command a bot fleet without reading source code (cf.
   `FUTURE-DIRECTIONS-2026-04-15.md` §10-11).

## The load-bearing architectural insight: dev mode vs live mode

**Core decision:** the same project needs fundamentally different
briefings at different lifecycle stages. A dashboard that uses
one view for all projects ignores that the commander's decisions
are different in each stage.

### Pre-launch / dev mode
The project is under construction. The commander's decisions are
about execution quality and velocity:

- Task queue depth (pending / bot-pickable / interactive-only)
- Recent cycle outcomes (verified / verified_weak /
  verification_failed counts over rolling window)
- Cycle velocity (cycles per day, trend)
- Current session status (running / idle, which project, which
  task, which cycle)
- Reviewer spend (today / week / month)
- Test count and coverage trend
- Verification gate evidence (hands-off violations caught,
  scope drift caught, silent failures surfaced)
- Launch-gate status (which gates closed, which remain)

### Live / production mode
The project ships; customers exist; money moves. The commander's
decisions are about business performance, not code velocity:

- Revenue (today / week / month, trend)
- Active user / customer counts
- Ad campaign status (active campaigns, CTR, ROAS, budget
  consumed)
- Spending limits and budget runway
- Uptime / error rate / response-time p95
- Support volume (open tickets, SLA status)
- Deployment status (last deploy, pending releases)

**A live project's dev metrics are still there but demoted** —
still available for drilldown, not on the front page. The
commander checks revenue first, not cycle velocity. Code work
continues (the bot still runs correctness cycles) but it's no
longer the headline.

### How a project transitions
Open question. Options:

- **Manual flag** in `projects.yaml`: `lifecycle: "dev" | "live"`
  (simple, explicit, commander-controlled)
- **Inferred** from signals: if revenue data is configured and
  non-zero, project is live; otherwise dev (automatic, fragile if
  data ingestion breaks)
- **Per-project state file**: `state/<id>/lifecycle.json` with
  commander-set stage + override rules (flexible, more config
  surface)

Recommendation: start with manual flag in projects.yaml. Add
automatic-by-signals later if the manual flag is consistently
wrong.

### How live-mode data gets ingested
Open question. The bot has no privileged access to Stripe,
Google Ads, Shopify, etc. Options:

- **Commander configures API keys per project** and the
  dispatcher polls (pulls on a schedule). BYOK applies — same
  pattern as Hard Rule 8 for LLM providers. No data ever leaves
  the user's machine.
- **Commander runs local connectors** (e.g., a small adapter
  script per data source) that writes to
  `state/<id>/live_metrics.json` on a schedule. Dispatcher reads
  from that file. Lower integration burden on GeneralStaff core.
- **Webhook receivers** bundled into `generalstaff serve` for
  push-based data (Stripe webhooks, etc.). More complex; requires
  the server to be reachable.

Recommendation: start with connector files (option 2). It's the
lowest-coupling, lowest-trust approach — the commander's existing
data pipelines write to a JSON file; the dispatcher reads it.
Cadence is the user's concern. API-key option (1) comes later if
connector proliferation becomes friction.

## The command-center shell (fleet rollup)

Regardless of per-project mode, the top of the dashboard is a
fleet-wide command center. Sections, in commander's-decision-
urgency order:

### 1. Situation
One-line rollup of the fleet's current state.
- Is a session running? Which project? Which cycle? How long?
- If idle: when was the last session? What was the outcome?
- Fleet: N projects registered, K with queue depth > 0, L in
  flight

### 2. Attention
Items that need the commander's decision or at least eyes. Empty
when nothing pressing — this section doesn't render if empty, to
avoid the "always a notification to clear" anti-pattern that
trains users to ignore alerts.

Candidates:
- Interactive-only tasks sitting in the backlog (waiting for Ray)
- Projects with drained queues (need seeding)
- Stranded `bot/work` commits that failed session-end merge
- Failed cycles in the last N hours
- Overspend or budget-exceeded conditions
- Stale project paths (git repo gone, etc.)

### 3. Fleet
One card per project. The card content depends on project
lifecycle mode (see "dev mode vs live mode" above). All cards
share: project name, lifecycle chip (dev / live / paused),
one-line status summary.

### 4. Actions
Commander-level controls.
- Queue a new task (project + title + optional priority/hints)
- Launch a session (budget + provider + optional project filter)
- Stop an active session
- Set / clear a `STOP` file for the fleet

### 5. Usage
Sidebar. LLM and other API spend across the fleet.
- Today / week / month totals
- Per-provider breakdown (Claude quota, OpenRouter $, Ollama
  free)
- Budget remaining if configured

## Phase 6 v1 scope

Ship the dev-mode dashboard. Defer live mode to Phase 7+.

**Included in v1:**
- Situation strip (session status, fleet rollup)
- Attention panel (render only if non-empty; interactive-only
  tasks + drained queues + stranded merges + recent failures)
- Fleet grid in dev mode only (project name, queue depth, last
  cycle outcome, 7-day trend as a simple sparkline or count)
- Actions: queue-task form, launch-session form
- Usage sidebar (today's reviewer spend from PROGRESS.jsonl
  scanning)

**Deferred to v2 / v3:**
- Live-mode metrics (revenue, ads, support, uptime)
- Live-mode data ingestion (connector files or API polling)
- Lifecycle mode switching (manual flag first, automatic later)
- Detailed per-cycle drilldown (use CLI `view dispatch-detail`
  for now)
- Charts/graphs beyond simple sparklines
- Push notifications (email, Telegram, etc.)
- Multi-user / role-based access

**Explicitly not a v1 concern:**
- Auth / multi-tenant (localhost-bound, single user)
- Remote access (Tailscale is the answer per
  FUTURE-DIRECTIONS §7)
- Theming (Kriegspiel aesthetic is Phase 6.5+ per
  UI-VISION-2026-04-15.md)

## Copy / voice guidelines

Every label should read like a subordinate briefing their
commander. Short, factual, informative. No jargon, no flair.

- ✅ "Your bot finished gs-253 successfully 4 minutes ago."
- ❌ "Cycle 202604191423 verified verdict."
- ✅ "Queue empty — gamr needs new tasks."
- ❌ "Project `gamr` has 0 pending bot-pickable tasks."
- ✅ "2 tasks are waiting on your decision."
- ❌ "2 interactive-only items in backlog."

The CLI keeps the terse / structural / machine-readable form
(that's its audience). The web UI translates. Both surfaces can
coexist because they serve different audiences.

## Accessibility as a first-class concern

Non-programmer audience includes people using assistive tech.
Ship it accessible from day 1 — retrofitting is always worse.

- Semantic HTML (`<section>`, `<article>`, `<nav>`, `<button>`,
  not `<div>` everywhere)
- Keyboard navigation for every action (no mouse-only controls)
- `aria-label` / `aria-live` where the semantics aren't obvious
- Color contrast WCAG 2.1 AA minimum (AAA for body text if
  feasible)
- Respect `prefers-reduced-motion` for any animations
- Never convey information with color alone (pair with icon
  or text)

## What this doc is NOT

- Not a UI-VISION-2026-04-15 replacement. That doc's theme /
  aesthetic decisions still stand for Phase 6.5+.
- Not a commitment to any specific framework — Phase 6 v1 stays
  vanilla HTML + `Bun.serve`; framework introduction is a
  future decision gated on concrete ergonomic pain.
- Not the data-ingestion design. Live-mode ingestion is an open
  question (see above). This doc says *what* the dashboard
  should eventually show; the *how* of getting live data in is
  a separate design pass.

## Open questions

1. **Lifecycle mode flag location.** `projects.yaml` (simplest),
   per-project `state/<id>/lifecycle.json` (most flexible), or
   inferred from signals (most automatic but brittle)?
2. **Live-mode connector contract.** What's the exact schema of
   `state/<id>/live_metrics.json`? Locked or open-ended?
3. **Sparkline / trend rendering.** Inline SVG, text art, or a
   lightweight library? Bias: text art is fine for v1; SVG for
   v2.
4. **"Launch session" form scope.** Just budget + provider, or
   also max-cycles / exclude-project / chain? Start minimal;
   add as asked.
5. **Session watching.** Does the dashboard poll for session
   state, or does the running session push events to the
   dashboard (SSE)? Poll-on-refresh for v1; SSE is Phase 6.5.
6. **Notifications.** When should the dashboard proactively
   surface something to the user who isn't looking at it? Out
   of scope for v1; revisit with Telegram integration if that
   ships again (MCP server appears to have flapped 2026-04-19).
